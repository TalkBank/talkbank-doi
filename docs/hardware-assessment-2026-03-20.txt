WHY THE UTM VM WAS SLOW
-------------------------

The VM config reveals the problem: it uses the QEMU backend with
virtio-ramfb-gl display. UTM has two backends:

  1. QEMU — software emulation layer on top of Apple Hypervisor. Supports
     x86 emulation (very slow) and ARM virtualization (faster). The
     virtio-ramfb-gl display driver is a generic fallback.

  2. Apple Virtualization Framework — native macOS hypervisor, much faster
     for ARM guests. Near-native performance for ARM Windows.

The VM IS aarch64 (ARM), which is good — it can use hardware
virtualization. But QEMU adds overhead compared to Apple's native
framework. The virtio-ramfb-gl display is also slower than the native
Virtualization.framework GPU passthrough.

Other likely issues:
  - CPUCount: 0 means "auto" in UTM/QEMU, which may not allocate enough
    cores. Should be set to 8+ explicitly.
  - 10 GB RAM is marginal for Windows 11 + Rust compilation + Python ML.
  - The Windows Insider Preview build may have debug/telemetry overhead.

SHARED FILESYSTEM: PITFALLS AND BEST PRACTICES
-----------------------------------------------

Q: Can I share the same git clone between macOS host and Windows VM?
A: DO NOT build from the same clone on both OSes. Here's why:

  1. Line endings: Git on Windows defaults to CRLF checkout. The Rust
     source, Python source, and .cha files are all LF. A shared clone
     will have constant line-ending conflicts.

  2. Build artifacts: Rust's target/ directory is platform-specific.
     A macOS target/ and Windows target/ in the same directory will
     collide and cause build failures.

  3. Symlinks: macOS uses symlinks freely; NTFS symlinks require
     elevated privileges and behave differently.

  4. File locking: Windows locks open files (DLLs, .exe). Rust builds
     on one OS will fail if the other OS has files locked.

INSTEAD, use one of these workflows:

  Option A: Git over LAN — separate clones, no GitHub round-trip (RECOMMENDED)
    - Host: ~/talkbank/ (your normal workspace, commit as usual)
    - VM: C:\Users\you\talkbank-tools\ (separate clone FROM the host)
    - Clone directly from the macOS host over the VM's virtual network:

        # In the Windows VM (one-time setup):
        git clone ssh://chen@192.168.64.1/Users/chen/talkbank/talkbank-tools C:\talkbank-tools
        git clone ssh://chen@192.168.64.1/Users/chen/talkbank/batchalign3 C:\batchalign3

        # Day-to-day: commit on macOS, then in the VM:
        git pull

    - No push to GitHub needed. The VM pulls directly from your macOS
      working tree over the local virtual network (sub-millisecond latency).
    - Each clone has its own target/, .venv, line endings, etc.
    - VMware Fusion's NAT networking makes the host accessible from the
      guest. The exact host IP depends on the network config — check with
      `ipconfig` in the VM or VMware's Virtual Network Editor. Often
      192.168.64.1 or the gateway shown in `route print`.
    - macOS SSH server must be enabled: System Settings → General →
      Sharing → Remote Login.

  Option B: Shared folder (read-only) + local build dir
    - Share the source tree read-only via VMware Shared Folders (HGFS)
    - In the VM, set environment variables:
        set CARGO_TARGET_DIR=C:\rust-target
        set CARGO_HOME=C:\cargo-home
    - Build from the shared source but write all artifacts locally
    - Configure git in the VM: core.autocrlf=false
    - Pro: no separate clone to maintain, always see latest saved files
    - Con: can't test uncommitted changes that require git operations;
      line ending issues may still surface in some tools
    - Best for quick "does this compile on Windows?" checks

  Option C: Push to GitHub (simplest but slowest)
    - Push to a branch on GitHub from macOS
    - Pull in the VM
    - Most overhead per iteration, but zero setup

  Option A is recommended for regular Windows testing. Option B is good
  for quick one-off build checks. Option C is the fallback when networking
  between host and guest is being difficult.

SCRIPTING VMWARE FUSION WITH vmrun
-------------------------------------

VMware Fusion ships with `vmrun`, a CLI for automating VM operations.
With VMware Tools installed in the guest, it can run commands inside the
VM, transfer files, and manage snapshots — all from the macOS terminal.

  Key commands:

    # VM lifecycle
    vmrun start /path/to/Windows.vmx
    vmrun stop /path/to/Windows.vmx
    vmrun suspend /path/to/Windows.vmx

    # Run commands inside the guest (requires VMware Tools)
    vmrun -gu <user> -gp <pass> runScriptInGuest /path/to/Windows.vmx \
      "C:\Windows\System32\cmd.exe" \
      "cd C:\talkbank-tools && git pull && cargo nextest run"

    # File transfer (host <-> guest)
    vmrun copyFileFromGuestToHost /path/to/Windows.vmx \
      "C:\talkbank-tools\test-results.xml" ./test-results.xml

    # Snapshots (clean-state testing)
    vmrun snapshot /path/to/Windows.vmx "dev-ready"
    vmrun revertToSnapshot /path/to/Windows.vmx "dev-ready"

  Recommended workflow:

    1. Set up VM once: install Windows, Rust, Python, uv, Git, VS Build
       Tools. Take a "dev-ready" snapshot.

    2. Automate the test cycle (e.g., a Makefile target):
       a. Revert to "dev-ready" snapshot (clean state)
       b. Start VM
       c. runScriptInGuest: git pull from macOS host over LAN
       d. runScriptInGuest: cargo nextest run (or cargo build, etc.)
       e. copyFileFromGuestToHost: retrieve test results
       f. Suspend VM

    3. For interactive-only tests (Tauri desktop app, SmartScreen, CLAN
       IPC), start the VM and work manually — but even then, vmrun can
       automate the build step so you only do visual verification.

  This is not as clean as GitHub Actions CI, but it means you can run
  `make test-windows` from your macOS terminal and get Windows test
  results without touching the VM GUI for most things.
