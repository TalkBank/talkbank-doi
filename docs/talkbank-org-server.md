## nginx Config Management

The nginx config is **generated, not hand-edited**. Hand edits get overwritten on
next deploy.

- **Source of truth:** `webdev/config.toml` + `webdev/src/templates/nginx.conf.j2`
- **Generate:** `cd webdev && uv run generate-web-confs`
- **Deploy:**
  ```bash
  sudo cp conf.d/talkbank.conf /etc/nginx/conf.d/
  sudo nginx -t && sudo systemctl reload nginx
  ```
- **Helper script:** `webdev/scripts/apply_talkbank_nginx.sh`

### Virtual hosts

1. **Main site** (`talkbank.org`) — root at `/var/www/web/talkbank-web/site/`
2. **www redirect** (`www.talkbank.org`) — 301 to `talkbank.org`
3. **Bank subdomain redirects** (`*.talkbank.org`) — 301 to `talkbank.org/{bank}`
4. **Psyling** (`/psyling/`) — alias to `/var/www/web/psyling-web/site/`
5. **CGI** (`/cgi-bin/*`) — FastCGI proxy to fcgiwrap unix socket
6. **Downloads** (`/downloads/`) — alias to `/var/www/downloads/`
7. **Data proxy** (`/{bank}/data/`, `/{bank}/data-orig/`) — reverse proxy to John's Node app on port 4000

### Security headers

HSTS, X-Content-Type-Options, X-Frame-Options, X-XSS-Protection,
Referrer-Policy. TLS 1.2/1.3 with modern ciphers. HTTP/2 enabled.

## mergerfs Setup

mergerfs presents the 24 split data repos as 18 flat per-bank directories for John's
app. Read-only union filesystem — zero extra disk space, changes from `git pull` are
immediately visible.

- **Install:** `sudo apt install mergerfs`
- **Setup script:** `deploy/scripts/setup_mergerfs_views.py`
- **Mount:** `sudo python3 setup_mergerfs_views.py mount`
- **Unmount:** `sudo python3 setup_mergerfs_views.py unmount`
- **Status:** `sudo python3 setup_mergerfs_views.py status`
- **Generate fstab entries:** `python3 setup_mergerfs_views.py fstab`

The script mounts `/home/macw/data/{repo}` → `/var/data/view/{bank}/`. For unsplit
banks (14 of 18), this is a 1:1 mount. For split banks (childes, ca, phon, homebank),
multiple repos are merged into one view.

Add fstab entries for persistence across reboots — see script output.

## Tailscale

Provides remote SSH access to the VM via the `talkbank` hostname.

### After OS upgrades

The Tailscale apt source includes the Ubuntu codename. After a major OS upgrade:

```bash
# Check current sources
ls /etc/apt/sources.list.d/tailscale*

# Fix for new codename (replace CODENAME, e.g., "resolute" for 26.04)
curl -fsSL https://pkgs.tailscale.com/stable/ubuntu/CODENAME.noarmor.gpg \
  | sudo tee /usr/share/keyrings/tailscale-archive-keyring.gpg >/dev/null
echo "deb [signed-by=/usr/share/keyrings/tailscale-archive-keyring.gpg] \
  https://pkgs.tailscale.com/stable/ubuntu CODENAME main" \
  | sudo tee /etc/apt/sources.list.d/tailscale.list

# Clean up upgrade leftovers
sudo rm -f /etc/apt/sources.list.d/tailscale.list.{migrate,disabled,distUpgrade}
```

## Disk Resize

The VM disk can be expanded via CMU Campus Cloud (vSphere). Brian requests the
additional space from CMU IT, then we expand the partition on the VM side.

**After CMU provisions more space:**

```bash
# Rescan the disk
echo 1 | sudo tee /sys/class/block/sda/device/rescan

# Expand partitions
sudo parted /dev/sda resizepart 2 100%
sudo parted /dev/sda resizepart 5 100%

# Expand LVM physical volume and filesystem
sudo pvresize /dev/sda5
sudo lvextend -l +100%FREE /dev/mapper/ubuntu--vg-ubuntu--lv
sudo resize2fs /dev/mapper/ubuntu--vg-ubuntu--lv

# Verify
df -h /
```

Full reference with example output: `docs/legacy/resize.md`

## OS Upgrade Checklist

When upgrading Ubuntu on this machine:

1. **Before upgrading:**
   - Note which services are running: `systemctl list-units --type=service --state=running`
   - Back up nginx config: `sudo cp /etc/nginx/conf.d/talkbank.conf ~/talkbank.conf.bak`

2. **During upgrade:**
   - If prompted about config files, keep the local version
   - Apache may be flagged for removal — this is fine, we don't use it

3. **After upgrading:**
   - Verify nginx: `sudo nginx -t && sudo systemctl status nginx`
   - Verify fcgiwrap: `systemctl status fcgiwrap && ls -la /var/run/fcgiwrap.socket`
   - Verify CGI: `ls -la /usr/lib/cgi-bin/morgra2jpg*.cgi`
   - Fix Tailscale apt source (see [Tailscale](#tailscale) section)
   - Clean up `.migrate`, `.disabled`, `.distUpgrade` files in `/etc/apt/sources.list.d/`
   - `sudo apt update && sudo apt upgrade`
   - Verify mergerfs mounts: `sudo python3 setup_mergerfs_views.py status`
   - Verify John's Node app: `systemctl status talkbank-data` (or whatever service name)

## Reproducibility Roadmap

The goal: anyone should be able to recreate this server from scratch, whether at CMU
or at a successor institution. We're progressing through four levels:

### Level 1: Documented (current)

This document. A human reads it and reproduces the setup manually. This is the minimum
for succession and what we have today.

### Level 2: Scripted (next step)

A single `setup-talkbank-org.py` script that:
- Installs all packages (`apt install`, `snap install`, Tailscale)
- Clones all web repos to `/var/www/web/`
- Clones all data repos to `/home/macw/data/`
- Sets up mergerfs mounts
- Deploys nginx config from `webdev/`
- Configures certbot
- Installs GitHub Actions runner
- Sets up John's Node app as a systemd service

Idempotent — safe to re-run. Gets us to "provision a fresh Ubuntu VM and run one
script."

### Level 3: Containerized (future)

Package the web-serving layer into Docker:
- **Container 1:** nginx + fcgiwrap + CGI scripts + static web content
- **Container 2:** John's Node app
- **docker-compose.yml** to orchestrate both + certbot sidecar
- Data repos and web repos mounted as volumes (not baked into images)

Benefits: portable across cloud providers, testable locally, version-pinned
dependencies, a successor at any institution can `docker compose up`.

Challenges:
- Certbot in Docker needs DNS-01 challenge (not HTTP-01) or an external reverse proxy
- GitHub Actions runner can't easily run inside a container
- Data repos (72+ GB) need volume mounts, not image layers

### Level 4: Infrastructure as Code (aspirational)

Terraform or Pulumi to provision the VM itself on any cloud provider. Combined with
Docker, this gives full "git clone + terraform apply + docker compose up"
reproducibility.

Not needed today, but worth noting as the north star. A successor who isn't at CMU
would need to provision their own VM somewhere — having the infrastructure definition
in code makes that possible without guessing at our setup.
